## 1. Проблема и контекст

### Ситуация
Зуботехническая лаборатория (Сургут + Москва, лаборатория у Эльдара). Поток 5–10 заказов в день на техника, 7–10 техников суммарно. Все заказы живут в **1С**, настроенной под лабораторию.

### Главная боль
Работы сдаются день-в-день с приёмом пациента или с опозданием. Заказы теряются, техники не видят приоритеты. Руководитель не может объективно оценить загрузку и дисциплину каждого техника.

### Почему предыдущее решение не сработало
Telegram-уведомления из 1С: захлебнулись объёмом (слишком много сообщений), сломались из-за VPN-блокировки в РФ. Техники перестали обращать внимание.

### Цель
> **Работы регулярно сдаются за 1–2 дня до приёма пациента.**  
> Руководитель видит метрику соблюдения сроков по каждому технику и может принимать кадровые решения.

|------|-----|----------------------|

## 3. Функциональные требования

### 3.1 Интерфейс техника (мобильный, PWA)

**Список заказов:**
- Показывает ТОЛЬКО заказы конкретного техника
- Сортировка по умолчанию: ближайший дедлайн сверху
- Карточка заказа содержит:
  - Название работы и тип (металлокерамика, циркон и т.д.)
  - Имя пациента
  - Клиника и врач
  - Дата сдачи (дедлайн) — крупно
  - Дата приёма пациента
  - Статус работы
  - Краткое описание / особые пожелания

**Цветовая подсветка срочности:**
- 🔴 Красный — дедлайн сегодня или просрочен
- 🟡 Жёлтый — дедлайн завтра
- 🟢 Зелёный — дедлайн через 2+ дня

**Действия техника:**
- Сменить статус заказа (Новый → В работе → Готово → Сдано)
- Добавить текстовый комментарий к заказу
- Получить push-уведомление при появлении нового заказа

**Авторизация:**
- Вход по PIN-коду (4–6 цифр) — просто и быстро
- Без логина/пароля — не нужно

---

### 3.2 Дашборд руководителя (веб, десктоп + мобиль)

**Обзор лаборатории:**
- Все активные заказы по всем техникам
- Фильтр по технику, городу, статусу, периоду
- Карточки заказов — аналогично интерфейсу техника

**Метрики по технику:**
- Всего заказов за период
- Сдано вовремя / с опозданием (%)
- Среднее время опережения дедлайна (в днях)
- График динамики за последние 30/60/90 дней

**Кадровый инструмент:**
- Таблица-рейтинг техников по % соблюдения сроков
- Возможность выгрузить данные (CSV или PDF — на потом)

---

### 3.3 Уведомления

- Push-уведомление в браузере при появлении нового заказа
- Утреннее напоминание (08:00) о заказах с дедлайном сегодня и завтра
- Без спама — максимум 2 уведомления в день на техника

## 4. Нефункциональные требования

| Требование | Детали |
|-----------|--------|
| **Доступность** | Работает без VPN в России |
| **Платформа** | PWA — открывается в браузере на телефоне (Android + iOS) |
| **Скорость** | Список заказов загружается < 2 сек |
| **Оффлайн** | Последний список заказов виден без интернета (кэш) |
| **Безопасность** | Техник видит только свои заказы, не видит чужие |
| **Надёжность** | Данные берутся из 1С — источник истины не меняется |

## 6. Стек технологий

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| **API-источник** | 1С HTTP-сервис | Уже есть, настраивается 1С-программистом |
| **Backend** | Python 3.11 + FastAPI | Простой, быстрый, много примеров, Claude пишет хорошо |
| **База данных** | PostgreSQL | Надёжная, бесплатная, хранит комментарии и историю |
| **Frontend** | React + PWA | Работает как приложение на телефоне без App Store |
| **Уведомления** | Web Push (браузерные) | Без Telegram, без VPN |
| **Хостинг** | VPS Timeweb (Россия) | Не блокируется, дёшево, без VPN |
| **Деплой** | Docker Compose | Запускается одной командой |

## 11. Процесс разработки: как мы работаем

Разработка ведётся совместно — руководитель лаборатории + Claude Code (AI-разработчик в Cursor).

**Роли:**
- **Руководитель** — принимает решения, тестирует на реальных данных, общается с 1С-программистом
- **Claude Code** — пишет весь код, объясняет решения, пушит документацию в GitHub

**Инструменты:**
- Cursor (IDE) — среда где работает Claude Code
- GitHub (`ArtSmile-ux/dental-lab-automation`) — весь код и документация
- Терминал — руководитель копирует и запускает команды по инструкции Claude

### Что происходит после согласования ТЗ

| Шаг | Действие | Кто |
|-----|----------|-----|
| **1** | Отправить ТЗ по API 1С-программисту, получить ответы на открытые вопросы (раздел 10) | Руководитель |
| **2** | Зарегистрироваться на Timeweb, арендовать VPS (Ubuntu 22.04, 1 CPU, 2 GB RAM) | Руководитель по инструкции Claude |
| **3** | Написать и задеплоить бэкенд FastAPI на VPS | Claude пишет, руководитель запускает команды |
| **4** | Проверить интеграцию с 1С: данные приходят корректно | Оба |
| **5** | Разработать PWA-интерфейс техника, протестировать на телефоне | Claude пишет, руководитель тестирует |
| **6** | Пилотный запуск с 1 техником — живой тест | Руководитель |
| **7** | Доработки по обратной связи, раскатка на всех техников | Оба |
| **8** | Дашборд руководителя с метриками | Claude пишет, руководитель проверяет |

**Принцип работы:** каждый шаг фиксируется в этом документе (обновляем статусы чеклистов), весь код и документация живут в GitHub.
